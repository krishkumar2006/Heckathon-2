# Implementation Plan: Todo App вЂ” Full-Stack Task Manager

**Branch**: `001-todo-app` | **Date**: 2026-02-27 | **Spec**: [spec.md](./spec.md)
**Input**: Feature specification from `/specs/001-todo-app/spec.md`

## Summary

Build a full-stack Todo application with JWT-based authentication, per-user task CRUD,
and Neon PostgreSQL persistence. The backend is a FastAPI service (Python) that verifies
JWT tokens on every endpoint and filters all task queries by `user_id`. The frontend is
a Next.js 16 App Router application (TypeScript + Tailwind) that authenticates via Better
Auth, stores a JWT, and routes all API calls through a single centralized client. Both
services share the same `BETTER_AUTH_SECRET` for token signing/verification.

**Language/Version**:
- Backend: Python 3.11+
- Frontend: TypeScript 5 / Node.js 20+

**Primary Dependencies**:
- Backend: FastAPI 0.110+, SQLModel 0.0.16+, python-jose[cryptography] 3.3+,
  python-dotenv 1.0+, psycopg2-binary 2.9+, uvicorn 0.27+

UI journey. No automated test suite in scope for hackathon.

**Target Platform**: Web application вЂ” modern desktop and mobile browsers

**Project Type**: Web application вЂ” two separate services (`frontend/` + `backend/`)

**Performance Goals**: API responses < 500ms for a single user's task list (в‰¤ 100 tasks).
Dashboard load < 2 seconds on first visit.

**Constraints**:
- Two separate `.env` files вЂ” never committed to git
- `BETTER_AUTH_SECRET` must be identical in both services
- No raw SQL вЂ” SQLModel ORM only
- No `any` types in TypeScript
- All API calls through `frontend/lib/api.ts` only
- No inline styles вЂ” Tailwind classes only

**Scale/Scope**: Single hackathon demo. Small user count (< 100). No pagination required.

### PLAN-04: Backend Auth Middleware

**Objective**: JWT verification middleware that all task routes will depend on.

**Tasks in order**:

1. Generate `BETTER_AUTH_SECRET`: `openssl rand -base64 32`
2. Set same secret in `backend/.env` AND `frontend/.env.local`
3. Create `backend/auth.py`:
   - `verify_token(token: str) в†’ dict`: decode with `jose.jwt.decode()`, algorithm HS256,
     return payload, raise `HTTPException(401)` on any failure
   - `get_current_user(credentials=Depends(HTTPBearer())) в†’ dict`: extract Bearer token,
     call `verify_token`, return user dict, raise `HTTPException(401)` on failure
4. Test middleware independently with valid + invalid tokens

**Checkpoint**: Valid token в†’ payload returned; invalid/missing token в†’ 401.

### PLAN-05: Backend CRUD Endpoints

**Objective**: All 6 task endpoints implemented, JWT-protected, user_id-filtered.

**Endpoint pattern** (every route):
```
1. current_user = Depends(get_current_user)         в†’ 401 if no/bad token
2. if current_user["sub"] != user_id в†’ 403          в†’ 403 if wrong user
3. SELECT/INSERT/UPDATE/DELETE WHERE user_id = user_id  в†’ data isolation
```

**Tasks**:
- `GET /api/{user_id}/tasks` вЂ” list with optional `status` filter (all/pending/completed)
- `POST /api/{user_id}/tasks` вЂ” create task (201)
- `GET /api/{user_id}/tasks/{task_id}` вЂ” get single (404 if not found)
- `PUT /api/{user_id}/tasks/{task_id}` вЂ” partial update (set updated_at UTC)
- `DELETE /api/{user_id}/tasks/{task_id}` вЂ” delete (204 No Content)
- `PATCH /api/{user_id}/tasks/{task_id}/complete` вЂ” flip completed bool

**Register router in main.py**: `app.include_router(tasks_router, prefix="/api")`

**Checkpoint**: All 6 routes visible at `/docs`; 401/403 tests pass; data in Neon.

### PLAN-08: Frontend API Client

**Objective**: Centralized API client that auto-attaches JWT to every backend call.

**Tasks in order**:

     - Handle 500 в†’ throw `Error("Server error, try again")`

     `updateTask`, `deleteTask`, `toggleComplete`

**Checkpoint**: Each function calls the correct endpoint with correct method and body.
